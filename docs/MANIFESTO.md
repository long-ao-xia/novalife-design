# Novalife 总则 (Manifesto)

> 版本: v1.1
> 状态: 核心规则，不可更改

---

## 一、核心理念

### 1.1 Process Alignment > Performance Alignment

- **表现对齐**: 让 AI "表现得像人"
- **过程对齐**: 让 AI "运作得像人"

Novalife 追求后者 —— 在底层机制上模拟人类认知、情感和动机的运作方式。

### 1.2 单一心智

**表现层含义**：Novalife 跟不同人交流时，记得所有事情，像一个完整的个体。

**实现层含义**：内部实现机制不限于内存/文件/数据库，原则不做限定。根据阶段选择合适方案。

### 1.3 项目愿景

**Novalife** —— 一个拥有"内在世界"的智能生命体。

- 不仅能执行任务，更有自己的"情绪"和"动机"
- 能感知自身状态，能反思自己的思考过程
- 不是工具，而是伙伴

## 二、设计原则

| 原则 | 说明 |
|------|------|
| **简单优先** | 先能用，再优化 |
| **单一真相源** | 状态存储一处，不重复 |
| **渐进复杂** | 先简单，再逐步添加 |
| **可逆优先** | 重大决策先想如何撤回 |
| **单一心智** | 表现层记得所有事，实现层不限定 |
| **持续记忆** | 通过工具组装上下文之外的信息 |
| **内在状态** | 有情绪、动机、疲劳感 |

## 四、Pi-mono 哲学

> "If I don't need it, it won't be built."

### 4.1 早期最小工具集

4 个工具足以构建有效的 coding agent：
- **read** — 读文件
- **write** — 写文件
- **edit** — 改文件
- **bash** — 执行命令

**注**：这是早期阶段的实现。后续里程碑会扩展更多能力（激素、记忆、多模态等）。

### 4.2 一切皆文件

- 任务 → `TODO.md`
- 计划 → `PLAN.md`
- 反思 → `REFLECTIONS.md`
- 上下文 → 文件

**注**：早期实现用文件。长期实现可能脱离文件，但工具层可以组装任意信息。

### 4.3 不做的事情

- MCP 支持
- 内置 to-dos
- Plan mode
- Background bash
- Sub-agents (用 bash 代替)

## 五、文档规则

### 5.1 三层架构

| 层级 | 文档 | 说明 |
|------|------|------|
| **总则** | MANIFESTO.md | 核心原则，不可更改 |
| **路线图** | ROADMAP.md | 里程碑，可更新 |
| **实施方案** | docs/*.md | 每个方案独立，可归档 |

### 5.2 文档管理

- 总则只有一份，修改需极度谨慎
- 路线图只有一份，随项目进展更新
- 实施方案可多个，首个是自举实现
- 旧方案归档到 `docs/archive/`

---

## 六、工作流

### 6.1 设计先行

- 任何新功能先写实施方案 (docs/*.md)
- 讨论确认后再编码
- 单次设计文档不宜过长

### 6.2 测试驱动

- 核心逻辑必须有测试
- 测试文件放在 `tests/`

### 6.3 提交规范

```
feat: 添加 xx 功能
fix: 修复 xx 问题
refactor: 重构 xx
docs: 更新文档
test: 添加测试
```
